## 1. The Modern Data Ecosystem

A network of interconnected, continually evolving entities:
- **Data** — available in many formats, structures, and sources
- **Enterprise data environment** — where raw data is staged, organized, cleaned, and optimized
- **End users** — business stakeholders, analysts, programmers who consume data

**Emerging technologies reshaping the ecosystem:** Cloud Computing, Machine Learning, Big Data.
> Cloud computing = what gives every enterprise limitless storage + high-performance computing at nominal cost.

### Data roles — who does what

| Role | Responsibility |
|---|---|
| **Data Engineer** | Extracts, integrates, and organizes data into repositories; converts raw data into usable data; builds data architectures |
| **Data Analyst** | Gathers, cleans, mines, and analyzes data to derive insights |
| **Data Scientist** | Uses analytics + engineering to predict the future from past data (ML/DL models) |
| **Business Analyst / BI Analyst** | Uses insights and predictions to drive business decisions |

## 2. The Four Types of Data Analytics

| Type             | Question                        | Example                                           |
| ---------------- | ------------------------------- | ------------------------------------------------- |
| **Descriptive**  | "What happened?"                | Monthly revenue reports, historical summaries     |
| **Diagnostic**   | "Why did it happen?"            | Drill-down / root-cause analysis                  |
| **Predictive**   | "What will happen next?"        | Forecasting from historical data and trends       |
| **Prescriptive** | "What should be done about it?" | Recommendations, likelihood of different outcomes |

💡 Memory hook: **D-D-P-P** — Describe the past, Diagnose the cause, Predict the future, Prescribe the action.

## 3. The Data Analysis Process

1. **Understand the problem and desired result** — "where you are" and "where you need to be"
2. **Set a clear metric** — what will be measured and how
3. **Gather data**
4. **Clean data**
5. **Analyze and mine data**
6. **Interpret results**
7. **Present findings** — communicate so it impacts decision-making

> [!note] Analysis vs. Analytics
> Often used interchangeably (including in this course). Technically: *analysis* = detailed examination of elements/structure (can be non-numeric); *analytics* = systematic computational analysis of data. This course treats them as the same.

## 5. Generative AI for Data Analysts (condensed)

**What it is:** AI models that create new content (text, images, etc.) by learning patterns from existing data.

**Key techniques:** GANs (generator vs discriminator networks) · VAEs (encode/decode to generate similar data) · Transformers (predict next token — GPT, BERT).

**Applications in data analytics:**
- **Data augmentation** — synthetic data when real data is scarce/imbalanced
- **Anomaly detection** — learn the "normal" distribution to spot outliers (fraud, security, QC)
- **Text/image generation** — auto product descriptions, marketing content
- **Simulation & forecasting** — generate potential outcomes from historical data
- Also: testing and development, enhancing visualizations, filling in missing data points

**Limitations/concerns:** doesn't always identify sources · hard to assess bias · realistic-sounding inaccuracies · plagiarism, deepfakes, fake news · trust and provenance issues.

## 7. Data Repositories, ETL & Big Data

### Repository types
- **Databases** — relational (RDBMS: **ACID-compliant**, SQL is standard, great for OLTP — small inserts/updates/deletes) or non-relational (NoSQL: key-value, **document-based** — record + data in one document, works well with analytics; column-based; **graph-based** — visualize/find connections)
- **Data Warehouse** — consolidates data into one comprehensive storehouse (e-commerce: analyze purchase trends; healthcare: integrate EHRs, claims)
- **Data Mart** — sub-section of a warehouse for one business function/team (marketing mart, cardiology mart)
- **Data Lake** — raw structured/semi/unstructured data in **native format**, tagged with metadata (clickstream, medical images, genomic data)
- **Big Data Stores** — distributed compute + storage for huge datasets
- NoSQL's emergence let marts/warehouses handle non-relational data too.

### ETL — Extract, Transform, Load
Automated process converting raw → analysis-ready data:
1. **Extract** from source locations
2. **Transform** — clean, enrich, standardize, validate
3. **Load** into destination repository

**Data Pipeline** = the entire journey source → destination (broader term, includes ETL).

### Big Data — the 5 V's
| V | Meaning |
|---|---|
| **Velocity** | Speed at which data accumulates |
| **Volume** | Scale of data |
| **Variety** | Diversity of type and sources |
| **Veracity** | Accuracy and conformity to facts |
| **Value** | Turning data into business value |

**Tools:** **Hadoop** (distributed storage + processing), Hive (warehouse layer on Hadoop), **Spark** (general-purpose engine — **complex analytics in real time**).

## 9. Data Wrangling (Munging)

**Workflow: Discovery → Transformation → Validation → Publishing** — and document everything (changes, reasons, current data quality).

### Transformation tasks
- **Joins** — combine columns from multiple tables · **Unions** — combine *rows* from multiple tables
- **Normalize** — clean the DB of unused/redundant data · **Denormalize** — combine tables into one for faster querying
- **Clean** — profile data, visualize to spot outliers, fix missing values, duplicates, irrelevant data, inconsistent formats, syntax errors
- **Enrich** — add data points that improve the analysis

### Handling common issues
- **Missing values:** filter out records / source the info / **impute** (calculate from statistical values)
- **Duplicates:** remove · **Syntax errors:** fix · **Outliers:** evaluate (not always errors!)
- **Irrelevant data** = data that doesn't fit the use case
- Sanity checks: summary statistics, min/max, logic checks. *There is no perfect data.*

**Tools:** Excel Power Query, OpenRefine (transforms to TSV/CSV/XLS/XML/JSON), Google DataPrep, **Watson Studio Refinery** (built-in discover/cleanse/transform), Trifacta Wrangler, Python (Jupyter, NumPy, pandas), R (dplyr, data.table, jsonlite).
Choosing depends on: data size, structures, cleaning capabilities, infrastructure, ease of use, learnability.

## 10. Statistics & Data Mining

**Statistics** = branch of mathematics for collection, analysis, interpretation, and presentation of quantitative data.

### Descriptive statistics
**Central tendency** — locating the center of a sample:

**Mean** = $\frac{\sum x}{n}$ — e.g. {2, 3, 7} → 12/3 = **4**

**Median** — middle value when sorted. Odd count → exact middle ({3, **5**, 9} → 5). Even count → average of two middles ({3, 5, 7, 10} → (5+7)/2 = **6**).

**Mode** — most frequent value ({2, **4**, **4**, 7, 9} → 4). Can be none, one, or multiple.

**Dispersion** — variability:

**Variance** — average squared distance from the mean:
- Population: $\sigma^2 = \frac{\sum (x - \mu)^2}{N}$
- Sample (most used): $s^2 = \frac{\sum (x - \bar{x})^2}{n - 1}$
- Low variability → consistent values; high → dissimilar values, likelihood of extremes.

**Standard deviation** = $\sqrt{\text{variance}}$ — same units as the data (unlike variance).

**Range** = Max − Min — e.g. {3, 5, 8, 12, 21} → 21 − 3 = **18**.

**Skewness** — whether the distribution is symmetric around the center or skewed left/right.

### Inferential statistics
Making generalizations about data: hypothesis testing, confidence intervals, regression analysis.

### Data mining
= **the process of extracting knowledge from data.** Techniques: classification · clustering · anomaly/outlier detection · association rule mining · sequential patterns · affinity grouping · decision trees · **regression (identifies the nature of the relationship between two variables — causal or correlational)**.

- **Trend** = general tendency of data to change over time · **Pattern** = discovery from data
- Tools: spreadsheets, **R (built for classification-type mining ops)**, Python (**pandas, NumPy**), IBM SPSS, Watson Studio, SAS.

## 11. Communicating Findings & Visualization

**Story + visualization + data.** Ask: Who is my audience? What matters to them? — **A presentation is not a data dump**: include only what addresses the business problem, in the language of the business domain.

**Structure:** reference your data, state your assumptions, organize the presentation, pick the best formats.
**Trust:** share your **data sources, hypotheses, and validations**.

### Chart selection
| Chart | Best for |
|---|---|
| **Bar chart** | Comparing related datasets or parts of a whole |
| **Column chart** | Side-by-side comparison; change over time |
| **Pie chart** | Breakdown of an entity into sub-parts (sums to 100%) |
| **Line chart** | **Trends** — value changing against a continuous variable |

**Dashboards** — bird's-eye view, organize reports/visualizations from multiple sources in one interface; easy to comprehend, enable collaboration, reports on the go.

**Goal of data visualization:** make information **easy to comprehend, interpret, and retain**.

**Tools:** spreadsheets (Google Sheets best for multi-user collaboration), Jupyter + Python libraries (**Matplotlib**), R-Studio/R-Shiny, IBM Cognos, Tableau, Power BI.

## 13. Final Project: Credit Card Fraud Case Study

**Scenario:** Data Analyst in a bank's credit card division; detect fraud early using historical transaction data. Most common fraud = stolen **credentials** (not the physical card) used online.

### Fraud red flags to watch (memorize — exam & interviews)
1. **Frequency change** — many transactions in a short span (e.g., minutes apart) vs. usual couple per month
2. **Value anomaly** — orders far above the user's average
3. **Bulk orders** of the same item with slight variations (color/size)
4. **Delivery change** — home address → in-store / warehouse / **P.O. Box** (untraceable)
5. **Mismatched IP address** — not from the billing address area

### Approach
1. Identify and gather relevant data: **cardholder details, transaction details, delivery details, location, network/IP**
2. Clean the data: missing values (IP, transaction value, age), inconsistent date formats (`15-5-20` vs `01 June 2020`), embedded currency symbols
3. Detect anomalies via descriptive analysis (patterns + deviations)
4. Visualize the trend to communicate findings

## 🎯 Interview Questions — JDA Path (Course 1 concepts)

Practice answering these out loud. Model answers included — personalize them with my own examples (NPower labs, fraud case study, freelance projects).

**Q1. Walk me through the four types of data analytics with a business example of each.**
> Descriptive tells us *what happened* — e.g., a monthly sales report. Diagnostic explains *why* — drilling into which campaign drove a sales dip or spike. Predictive uses historical data to forecast *what's next* — like predicting churn. Prescriptive recommends *what to do* — e.g., a recommendation engine suggesting price changes. In practice they build on each other: you can't prescribe until you can describe.

**Q2. Describe the data analysis process end-to-end.**
> Start by understanding the problem and the desired outcome — where we are vs. where we need to be — and agree on a metric for success. Then gather data from the right sources, clean it, analyze and mine it for patterns, interpret the results in business terms, and present findings in a way that drives a decision. Documentation throughout for repeatability.

**Q3. What's the difference between a data analyst, data engineer, and data scientist?**
> Engineers build the pipelines and repositories — they turn raw data into usable data. Analysts gather, clean, and analyze that data to produce insights about what happened and why. Scientists build predictive models on top of it. Coming from 10+ years of software development, I sit comfortably at the analyst/engineer boundary — I can write my own SQL and Python rather than waiting on someone else.

**Q4. Structured vs. semi-structured vs. unstructured data — give examples and where each is stored.**
> Structured fits rows and columns with a rigid schema — transactional records, spreadsheets — stored in relational databases. Semi-structured has organizational properties but no rigid schema — emails, JSON, XML — often in NoSQL stores. Unstructured has no predefined model — video, images, social feeds — stored in NoSQL databases or data lakes in native format.

**Q5. Explain the difference between a data warehouse, a data mart, and a data lake.**
> A warehouse consolidates cleaned, structured data from across the organization into one analytical store. A mart is a focused slice of that warehouse for one team — marketing or finance — so they only see what's relevant. A lake stores raw data of any structure in its native format, cheap and flexible, typically for data science experimentation before structure is imposed.

**Q6. What is ETL? How does it differ from a data pipeline?**
> ETL extracts data from sources, transforms it — cleaning, standardizing, enriching, validating — and loads it into a destination repository. A data pipeline is the broader concept: the entire journey from source to destination, which usually includes ETL as a step.

**Q7. What are the 5 V's of Big Data?**
> Volume (scale), Velocity (speed of accumulation), Variety (diversity of types and sources), Veracity (accuracy and trustworthiness), and Value (the business worth extracted). Veracity is the one analysts live with daily — biased or noisy data kills an analysis faster than small data.

**Q8. How do you handle missing values in a dataset?**
> It depends on the mechanism and the use case. Options: remove the records if they're few and random; go back to the source to fill them; or impute using statistical values like the mean or median. In the IBM fraud case study, missing IP addresses were actually informative — a missing value can itself be a signal, so I document rather than blindly delete.

**Q9. What is an outlier and how do you deal with one?**
> An extreme value at either end of a distribution. First rule: investigate before deleting — an outlier can be a data-entry error, or it can be the finding itself. In fraud detection, the outliers *are* the fraud. I use summary statistics, min/max checks, and visualization to spot them, then decide based on the business context.

**Q10. Mean, median, mode — when would you use median over mean?**
> Mean is the arithmetic average; median is the middle sorted value; mode is the most frequent. Use the median when the distribution is skewed or has outliers — e.g., household income or house prices — because a few extreme values drag the mean but leave the median stable.

**Q11. What's the difference between primary, secondary, and third-party data?**
> Primary comes directly from the source — internal CRM, HR systems, my own surveys. Secondary is retrieved from external existing sources. Third-party is purchased from aggregators. Each step away from the source raises questions of quality, licensing, and privacy compliance.

**Q12. How would you detect potentially fraudulent credit card transactions?** *(← my final project — own this story)*
> I'd start descriptive: establish each customer's baseline — frequency, average value, usual delivery address and IP. Then flag anomalies where multiple signals shift together: transactions minutes apart vs. a normal couple per month, values 20–40x the baseline, a sudden switch from home delivery to a P.O. Box, and a mismatched or missing IP. In the IBM capstone I identified exactly that stacked pattern, after first cleaning missing values and standardizing inconsistent date formats so the timeline analysis was reliable.

**Q13. How do you make stakeholders trust your analysis?**
> Share the process, not just the conclusion: data sources, assumptions, hypotheses, and validations. Speak the language of their business domain, structure the presentation around their decision, and never do a data dump — only what addresses the problem.

**Q14. Which chart would you pick to show a trend over time, and why?**
> A line chart — it's designed for showing how a value changes against a continuous variable like time. Bar/column for comparisons, pie for proportional breakdowns of a whole, but trends belong to lines.
